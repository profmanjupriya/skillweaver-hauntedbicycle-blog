Pulling One Thread in Someone Else's Codebase

What a scoring bug taught us about reading code we didn't write, calmly.

Bricen Feb 3, 2026 9 min read
In this post
  1. The report
  2. Resisting the urge to rewrite
  3. Following one thread
  4. The actual bug
  5. The habit worth keeping

A support ticket came in from an instructor: a student's final score didn't match what the item-by-item breakdown implied. Add up the points and you got one number; the dashboard showed another, two points lower. Small enough to miss, large enough to matter if you're the student it happened to.

The scoring module was written before I joined the project, by someone no longer on the team, with no tests covering the specific aggregation path involved. My first instinct, the instinct I think most of us have, was to just rewrite the function so I could trust it. That instinct is usually wrong, and this post is about why.

The report

The scoring pipeline in Skill Weaver takes a set of item responses, each with a difficulty weight from the adaptive engine, and aggregates them into a final score using a weighted formula tied to the underlying item response theory model. It's the kind of code that's correct in a way that's hard to verify by inspection, because the correctness lives in a formula, not in control flow you can step through and eyeball.

Rewriting code you don't understand doesn't remove the risk. It just moves the risk somewhere you can't see it yet.

Resisting the urge to rewrite

Unfamiliar code with no tests is uncomfortable to work in, and rewriting it feels like progress because you end up with something you understand. But a rewrite done under time pressure, without first understanding why the original was shaped the way it was, tends to fix the bug you know about and introduce two you don't. So the actual first step was slower than it wanted to be: reconstruct what the function was supposed to do before touching how it did it.

Following one thread

Instead of reading the whole scoring module top to bottom, I picked one thread: the specific student, the specific item set, and traced only the values that fed into their final number. That meant adding temporary logging at each transformation step and re-running the aggregation against that one student's actual response data, rather than synthetic test input.

// temporary trace added at each stage of aggregation
console.log('raw item scores', itemScores);
console.log('difficulty weights', weights);
console.log('weighted sum', weightedSum);
console.log('normalization factor', normFactor);
console.log('final score', weightedSum / normFactor);

This surfaced something immediately: the weighted sum was correct, but the normalization factor was being computed from the count of items attempted, while the weighted sum itself included a partial-credit adjustment for one item that had been flagged and re-scored after submission. The numerator had been updated to reflect the re-score. The denominator hadn't.

The actual bug

The normalization factor was calculated once, early in the pipeline, and cached on the response object before the re-scoring step ran later in the same request. Re-scoring updated the numerator in place but never invalidated that cached denominator. Under normal conditions, without a re-score, this was invisible, because both values were computed from the same unmodified data. It only surfaced for the specific case of an item flagged for manual review after initial scoring, which is exactly what had happened here.

The fix was small: recompute the normalization factor after any re-scoring step, instead of trusting a value cached earlier in the request. But finding that required understanding the order operations ran in, which required not rewriting the function before I'd traced it.

The habit worth keeping

A few things I'm trying to keep doing after this one:

  • Trace one real, specific case before making any structural change to unfamiliar code.
  • Treat "this is cached" as a question, not a fact, especially near anything with a manual override or re-scoring path.
  • Write the missing test for the exact case you just found, not a generic one, so the next person gets the specific regression protection this bug earned.

The rewrite instinct isn't wrong because rewrites are bad. It's wrong because it skips the step where you find out what you're actually rewriting.
