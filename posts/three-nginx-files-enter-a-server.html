Three Nginx Files Enter a Server

A deployment postmortem in three configuration files, one of which was lying to us the entire time.

Jesus Jan 20, 2026 8 min read

On a Tuesday, roughly one in twenty requests to the Skill Weaver scoring endpoint started timing out. Not failing, not erroring, just hanging until the client gave up. The service itself was healthy. CPU and memory were flat. The database had capacity to spare. Which meant the problem was somewhere between the internet and the application, and on our setup, that means Nginx.

The symptom

Intermittent timeouts are the worst kind of bug to reproduce on purpose, so we started by just reading. Three files stood between a request and our application: the main server block, an upstream block defining the application pool, and an include for shared proxy settings pulled into both. Any one of the three could plausibly have caused this.

The bug wasn't in the file we suspected. It was in the file we'd stopped reading because we trusted it.

File one: the one we suspected

The server block was the newest of the three, edited two weeks earlier to add a route for a new endpoint. Fresh changes are always suspect first, so we diffed it line by line against version control. Nothing unusual: a new location block, correctly scoped, no proxy settings overridden. This file was innocent.

File two: the one that looked fine

The upstream block defined our application pool with a simple round-robin configuration across three backend processes. This is the kind of file that looks correct at a glance because it's short, and short files get less scrutiny than they deserve. We checked it anyway: three servers listed, no weighting, no max_fails override. Also innocent, or so it seemed at the time.

File three: the one that was lying

The shared proxy include was the oldest file in the stack, written before either of us had joined the project, and it's the one nobody reads during a normal review because it never changes. It set proxy timeouts, buffer sizes, and headers, and had been copy-pasted from a template with a comment reading # standard config, don't need to touch.

Buried in that file was a proxy_next_upstream directive configured to retry a failed request against the next backend in the pool, but with no cap on total attempts and a timeout value that compounded across retries instead of resetting. Under normal load this was invisible. Under a specific pattern of slow requests to one backend, it meant a single client request could silently retry against all three backends in sequence, each one waiting out a nearly-full timeout window before failing over, adding up to a multi-minute hang that looked, from the client's side, like nothing was happening at all.

# before — retries with no cap, timeouts compound across attempts
proxy_next_upstream error timeout http_502 http_503;
proxy_next_upstream_timeout 0;
proxy_read_timeout 60s;

# after — bounded retries, explicit total timeout
proxy_next_upstream error timeout http_502 http_503;
proxy_next_upstream_tries 2;
proxy_next_upstream_timeout 15s;
proxy_read_timeout 8s;

The fix wasn't complicated once we found it. Cap the number of retry attempts, cap the total time spent retrying, and lower the per-attempt read timeout so a single slow backend fails fast instead of eating the whole budget. The hard part was noticing that the file we'd been treating as inert configuration was the one actually shaping behavior under load.

What we changed

  • Added explicit bounds to every retry directive in shared config, not just this one.
  • Added a comment above the file explaining what it actually controls, replacing the comment that told people not to touch it.
  • Added a synthetic slow-backend test to our deploy checklist, so a compounding timeout like this shows up before production does.

The lesson wasn't really about Nginx. It was that "standard config, don't need to touch" is exactly the sentence that should make you touch it.
